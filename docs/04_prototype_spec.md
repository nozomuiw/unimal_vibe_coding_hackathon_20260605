# 04. プロトタイプ技術仕様

## スコープ

- **形態**: Webアプリ（チャットUI）。1台の端末（スマホ/PCブラウザ）をAさん・Bさんが交代で使う
- **目的**: 一連の体験（INTAKE → HEARING → TRIAL → VERDICT）が完走できること
- **やらないこと**: 認証、課金、履歴の永続化（DB）、別端末参加、本番運用品質のインフラ

## 技術スタック（想定）

| レイヤー | 技術 | 備考 |
|---|---|---|
| フロントエンド | Next.js (App Router) + TypeScript + Tailwind CSS | チャットUI、フェーズごとの画面切替 |
| バックエンド | Next.js API Routes（同一リポジトリ） | セッション状態管理とLLM呼び出し |
| LLM | Claude API（`@anthropic-ai/sdk`） | 詳細は後述 |
| 状態保存 | サーバーメモリ上の `Map<sessionId, CourtSession>` | プロトタイプはプロセス再起動で消えてよい。ブラウザ側は sessionId のみ保持 |
| ストリーミング | SSE（SDKの `messages.stream()` をそのまま中継） | 代理人・裁判官の発話をタイプライター表示 |

APIキーはサーバー側の環境変数 `ANTHROPIC_API_KEY` でのみ扱い、クライアントには一切渡さない。

## アーキテクチャ概要

```
┌─ Browser ──────────────────────────────┐
│  チャットUI / フェーズ画面 / 受け渡し画面  │
└───────────────┬────────────────────────┘
                │ POST /api/session/:id/turn （SSEで応答）
                ▼
┌─ Next.js Server ───────────────────────┐
│  State Machine（フェーズ管理・ターン上限） │
│  ├─ CourtSession（事件記録・対話ログ）    │
│  ├─ Role Prompt Builder（役割別プロンプト）│
│  └─ Safety Gate（エスカレーション判定）    │
└───────────────┬────────────────────────┘
                │ messages.stream() / messages.create()
                ▼
         Claude API（claude-opus-4-8）
```

### マルチエージェントの実現方法

A代理人・B代理人・裁判官・書記官は、**同一モデルに対するシステムプロンプトの切り替え**で実現する（エージェントごとに別モデルやプロセスを持たない）。

- 各ロールの呼び出しは独立した `messages.create()`。会話履歴はサーバー側の `CourtSession` が一元管理し、ロールごとに「そのロールが見てよい情報」だけを組み立てて渡す
- 裁判官キャラクターは、裁判官システムプロンプトのテンプレートにキャラ定義（[03_characters.md](03_characters.md) の YAML）を展開して生成する。選定はサーバー側の乱数

### 情報の見え方（コンテキスト分離）

| ロール | 渡すコンテキスト |
|---|---|
| A代理人（ヒアリング時） | 事件概要 + Aとの対話ログのみ |
| B代理人（ヒアリング時） | 事件概要 + Aの主張**サマリー** + Bとの対話ログ |
| 代理人（法廷時） | 事件概要 + 自陣の承認済みサマリー + 法廷の公開ログ |
| 裁判官 | 事件概要 + 双方の承認済みサマリー + 法廷の公開ログ（**生のヒアリングログは渡さない**） |

ヒアリングの生ログを裁判官・相手方代理人に渡さないことで、[02_flow.md](02_flow.md) のプライバシー設計をプロンプトレベルでも担保する。

### セッション状態（CourtSession）

```typescript
interface CourtSession {
  id: string;
  phase: "INTAKE" | "JUDGE_SELECT" | "HEARING_A" | "HANDOFF"
       | "HEARING_B" | "TRIAL" | "RE_HEARING" | "VERDICT"
       | "CLOSING" | "ESCALATION";
  caseTitle: string;              // 事件名
  parties: { a: string; b: string };
  judge: JudgeProfile;            // ランダム選定されたキャラ
  hearings: {
    a: { log: Message[]; summary?: string; approved: boolean };
    b: { log: Message[]; summary?: string; approved: boolean };
  };
  courtLog: Message[];            // 法廷の公開ログ（TRIAL以降）
  reHearingCount: number;         // 最大2
  verdict?: VerdictResult;
}
```

フェーズ遷移・ターン上限・再聞き取り回数の制御は**すべてサーバー側のコード**（決定的なステートマシン）で行い、LLMに進行管理を任せない。LLMが返すのは「発話」と「判定の構造化出力」のみ。

## Claude API 利用設計

### モデル選定

**全ロールで `claude-opus-4-8` を使用する**（2026-06時点の最新・最上位モデル）。

| モデル | Input $/1M | Output $/1M | 本サービスでの位置づけ |
|---|---|---|---|
| **Claude Opus 4.8** (`claude-opus-4-8`) | $5.00 | $25.00 | **採用**。判決の説得力・キャラの演じ分け・日本語の説諭の質がプロダクト価値そのものなので、全ロールで使用 |
| Claude Sonnet 4.6 (`claude-sonnet-4-6`) | $3.00 | $15.00 | コスト削減オプション（ヒアリングのみ Sonnet 化等）。品質比較の上で判断する将来課題 |
| Claude Haiku 4.5 (`claude-haiku-4-5`) | $1.00 | $5.00 | 完了判定などの内部判定のみの切り出し候補。同上 |

判決・説諭はこのサービスの中核体験であり、プロトタイプ段階でモデル起因の品質問題と設計起因の問題を切り分けられなくなるのを避けるため、まず最上位モデルで体験を成立させる。コスト最適化はその後の課題とする。

### 呼び出しパラメータ方針

| 用途 | 設定 |
|---|---|
| 代理人・裁判官の発話（全般） | `messages.stream()` でストリーミング。`max_tokens: 4096` 目安 |
| 判決文の生成 | `thinking: { type: "adaptive" }` + `output_config: { effort: "high" }`。複数論点の整合的な判断に思考を使わせる |
| 内部判定（完了判定・判定可否） | `output_config.format`（structured outputs, `json_schema`）で構造化出力を強制。非ストリーミング、`max_tokens: 1024` |
| サンプリング | Opus 4.8 は `temperature` / `top_p` / `top_k` 非対応（送ると400）。出力の揺らぎはプロンプトで制御 |

### 構造化出力（内部判定）

LLMの判定をコードの分岐に使う箇所は、すべて structured outputs で受ける:

1. **ヒアリング完了判定**（毎ターン）: [03_characters.md](03_characters.md) のスキーマ。`hearing_complete` と `safety_concern` をステートマシンが参照
2. **裁判官の判定可否**（ラウンド末）:
   ```json
   {
     "verdict_possible": false,
     "missing_points": ["購入前の家計ルールの有無"],
     "questions_for_a": ["…"],
     "questions_for_b": ["…"]
   }
   ```
3. **判決データ**（判決文の生成と同時に取得）:
   ```json
   {
     "ratio_a": 70,
     "ratio_b": 30,
     "proposals": ["2万円以上の買い物は事前に一報を入れる"],
     "safety_concern": "none"
   }
   ```
   判決「文」はキャラ口調のテキストとしてストリーミングし、判決「データ」（サマリーカード用）は別呼び出しで構造化取得する

### プロンプトキャッシュ

会話が積み上がる設計のため、プロンプトキャッシュを前提に組む:

- ロールごとのシステムプロンプト（キャラ定義含む）は**フリーズして固定**。日時・セッションIDなどの可変値を入れない
- 事件記録（事件名・サマリー）→ 対話ログの順で安定したプレフィックスを構成し、最新ターンの末尾に `cache_control: { type: "ephemeral" }` を置く
- 同一ヒアリング内の連続ターン（5分以内の応答が前提）はキャッシュヒットが期待でき、入力コストを大きく下げる

### セーフティゲート

- ヒアリング完了判定の `safety_concern` フィールドが `none` 以外を返したら、ステートマシンが `ESCALATION` に遷移
- 加えて全ロールのシステムプロンプトに対象外領域（[01_concept.md](01_concept.md)）と「法的助言をしない」制約を明記
- `stop_reason: "refusal"` を受けた場合も `ESCALATION` 扱いとする

## コスト試算

### 前提（1セッションあたりの呼び出し）

| フェーズ | 呼び出し内容 | 回数 |
|---|---|---|
| INTAKE / JUDGE_SELECT | 事件名生成 + 裁判官登場 | 2 |
| HEARING（A+B） | 発話 + 完了判定 × 10往復 × 2人 | 40 |
| TRIAL（1ラウンド） | 裁判官2 + 代理人4 | 6 |
| RE_HEARING（1回 = 2人×3往復×2呼び出し） | 発話 + 完了判定 | 12 |
| VERDICT | 判決文 + 判決データ | 2 |

入力は会話の積み上がりにより1呼び出しあたり平均 3〜6Kトークン、出力は発話 250〜400 / 判定 100 / 判決 1.5K トークン程度と仮定。

### 試算結果（Opus 4.8、1ドル150円換算）

| シナリオ | 入力計 | 出力計 | キャッシュなし | キャッシュあり(※) |
|---|---|---|---|---|
| 最短（再聞き取りなし） | 約185K | 約11K | 約$1.2（約180円） | 約$0.5（約75円） |
| 標準（再聞き取り1回） | 約270K | 約15K | 約$1.7（約260円） | 約$0.7（約105円） |
| 最長（再聞き取り2回） | 約360K | 約19K | 約$2.3（約350円） | 約$1.0（約150円） |

※ 入力の約7割がキャッシュ読み取り（0.1倍課金）になると仮定した概算。

**注意**: 上記は机上の試算であり未検証。実装後に `usage`（`input_tokens` / `cache_read_input_tokens` / `output_tokens`）を必ずログして実測で補正すること。聞き取りが長引くユーザーでは上振れする。

### コストが問題になった場合の削減レバー（優先順）

1. プロンプトキャッシュのヒット率改善（システムプロンプトの可変値混入を監査）
2. 完了判定（毎ターンの構造化出力）を `claude-haiku-4-5` に分離
3. ヒアリング上限ターンの引き下げ（10 → 7）
4. ヒアリングロールのみ `claude-sonnet-4-6` 化（品質比較の上で判断）

## 開発マイルストーン

| Step | 内容 | 完了条件 |
|---|---|---|
| 1 | ステートマシン + CourtSession + Claude API疎通 | CLIまたはAPI経由で全フェーズを通しで実行できる（UIなし） |
| 2 | 役割プロンプト一式（代理人・裁判官5キャラ・書記官） | 同一事案で裁判官5キャラの判決の味が変わることを目視確認 |
| 3 | チャットUI + SSEストリーミング + HANDOFF画面 | ブラウザで2人交代の通し体験ができる |
| 4 | 構造化出力による判定の安定化 + セーフティゲート | 完了判定・判定可否・エスカレーションが安定して分岐する |
| 5 | 演出（ルーレット、判決カード）+ usage計測 | 社内/家庭内テストに出せる状態。コスト実測値の取得 |

## 評価方法（プロトタイプの検証項目）

1. **完走率**: テストペアが20〜30分で VERDICT まで到達できるか
2. **聞き取りの質**: ヒアリング後のサマリーを本人が「言いたいことが入っている」と承認できるか（修正回数を記録）
3. **判決の受容性**: 負けた側が「納得はしないが受け入れられる」と答えるか
4. **キャラの効き**: 裁判官の違いを体験者が言い当てられるか
5. **コスト実測**: 1セッションあたりの実コストが試算（標準シナリオ ±50%）に収まるか
