# Claude Design プロンプト — パターンA：全8画面フルデモ

> **目的**: 以下のプロンプトを Claude（Artifacts / design モード）に貼り付け、プレゼン用の**台本どおり進むクリッカブルプロトタイプ**「家庭内裁判所」を制作する。UI文言は**日本語**。指示書本文も**日本語**（英語版は `06_design_prompt_full_en.md`）。
>
> **推奨ワークフロー**: **段階プロンプト（Step 0→5）**を同一スレッドで順に貼る。時間がない場合のみ**一発版マスタープロンプト**を使用。

---

## このドキュメントの使い方

1. Claude Artifacts の新規会話を開始（サイドバー → Artifacts）。
2. マスター仕様（下記）を会話の**先頭に pin** する。
3. **Step 0** を貼り、確認後、**Step 1 → 5** を**同じスレッド**で順に貼る。
4. 裁判官イラストが用意できたら、`JUDGE_IMAGES` に URL または data URI を貼る（Step 4）。
5. 修正は**付録：反復のコツ**に従い、全面書き直しは避ける。

## 共有ブロック（最初に pin）

### マスター仕様（会話先頭に pin）

```
React 単一ファイルコンポーネントのアーティファクトを作成：台本どおり進むクリッカブルデモ「家庭内裁判所」— 夫婦の家庭内紛争を AI 裁判というエンタメ形式で整理・決着させるアプリ（提訴→聞き取り→法廷→判決）。1台の端末を2人で交代使用。LLM/API 呼び出しなし。以下のシナリオデータをすべてハードコード。モバイルファースト（最大幅 ~430px）、UI は日本語、ポップでエンタメ性のあるビジュアル。状態管理は useState/useReducer のみ（localStorage 不可）。ライブラリ：Tailwind CSS、lucide-react のみ。画像は JUDGE_IMAGES マップ（URL または data URI）経由。全画面に免責表示：「本サービスはエンターテインメントであり、法的助言・カウンセリングではありません。」
```

### プロダクトコンテキスト

**解決したい課題**: 家事分担・お金・子育てなどの小さな争いには、中立な第三者がいない。友人は中立ではない。カウンセリングは大げさすぎる。

**提供価値**:
- **中立性**: AI は利害関係がない。代理人は各当事者の味方。裁判官は社会通念で判断
- **言語化の支援**: 代理人 AI が事実・感情・要望の整理を手伝う
- **エンタメ性**: 「裁判」メタファー＋キャラ裁判官が深刻な対立をゲーム的儀式に変換
- **決着**: 必ず判決で一区切りつける

**設計思想**:
- 裁判官は**ランダム選定**され、意図的なキャラバイアス（±15pt 上限）を持つ — これは仕様。敗訴側は「あの裁判官だから」と受け流せる
- 代理人は聞き取り中は常に**依頼人の味方** — 説教しない
- プライバシー: 生の聞き取りログは相手に見せない（承認済みサマリーのみ法廷へ）

### ビジュアル方針

**トーン**: ポップ、明るく、親しみやすい — モバイルゲーム／キャラチャットアプリ風。裁判の厳かさと遊び心の両立。

**ベンチマーク参考**（これらをミックス）:
- Duolingo 風のフレンドリーなカードと丸ボタン
- ビジュアルノベル／LINE 風のキャラ吹き出し
- ガチャ／ルーレットの裁判官登場演出
- Stripe 風のきれいなカードレイアウト（判決サマリー）

**役割色**（CLI プロトタイプに対応）:
| 役割 | 色 | 用途 |
|------|-----|------|
| 書記官 | グレー／ニュートラル | システムメッセージ |
| A側代理人 | シアン（`#06b6d4` 系） | 美咲の代理人 |
| B側代理人 | マゼンタ（`#d946ef` 系） | 健太の代理人 |
| 裁判官 | 琥珀（`#f59e0b` 系） | 裁判官の発話 |

**タイポグラフィ**: クリーンなサンセリフ。事件名は太字でやや儀式的に。

**禁止**: 裁判官の似顔絵イラストを生成しない。`JUDGE_IMAGES` スロットまたはイニシャルプレースホルダーのみ。

### アーティファクトサンドボックス制約（必須）

| ルール | 内容 |
|--------|------|
| 種別 | React 単一ファイルコンポーネント |
| 状態 | `useState` / `useReducer` のみ — **localStorage / sessionStorage 不可** |
| ネットワーク | 外部 API・fetch・LLM **不可** |
| ライブラリ | Tailwind CSS、lucide-react — jQuery、axios、lodash は避ける（描画失敗の可能性） |
| 画像 | **ローカルパス不可**。コード先頭の `JUDGE_IMAGES` マップ（URL または data URI）。`null` 時は色付き円＋イニシャル |
| TOC | ファイル先頭にセクション一覧コメント |
| エッジケース | 画像未設定、長文日本語の折り返し |

### キャラクター画像素材

```javascript
// アーティファクトコードの先頭に配置 — 後から URL または data URI を差し込む
const JUDGE_IMAGES = {
  judge_ooka: null,      // 大岡 忠相 — 肖像 1:1、最小 256×256
  judge_logica: null,
  judge_mitsuko: null,
  judge_iwaya: null,     // 巌谷 鉄五郎 — 本デモで使用
  judge_luna: null,
};
// null 時：色付き円＋名前の最初の漢字。AI 似顔絵は生成しない。
// 書記官・代理人：lucide アイコンのみ。
```

**配置箇所**: ルーレットのコマ、裁判官紹介カード、法廷の発話者アイコン、判決画面ヘッダー。

## 段階プロンプト（メイン — 順に貼る）

### Step 0: キックオフ

```
pin したマスター仕様と共有ブロックをもとに、「家庭内裁判所」のプレゼン用プロトタイプを作りたい。

コードを書く前に：
1. これが**台本どおりの React アーティファクト**（API なし、localStorage なし）であることを確認してください。
2. 実装する**8画面**とフェーズのステートマシンを一覧してください。
3. 不明点があれば**最大2問**まで質問してください。

まだコードは書かないでください。Step 1 の「go」まで待ってください。
```

### Step 1: 骨組み（レイアウト＋遷移のみ）

```
Step 1 — 骨組みのみ作成。

React 単一ファイルアーティファクト：
- フェーズ: TOP → INTAKE → JUDGE_SELECT → HEARING_A → HANDOFF → HEARING_B → TRIAL → VERDICT → CLOSING（+ ESCALATION はデモメニューから参照可能）
- モバイルファースト（~430px 中央）、ヘッダーに事件名＋フェーズ表示
- フッター免責：「※ 本サービスはエンターテインメントであり、法的助言・カウンセリングではありません。」
- デザイントークン：ポップ配色、役割色（書記官グレー、A=シアン、B=マゼンタ、裁判官=琥珀）
- 各画面はプレースホルダー＋「次へ」ボタン
- ファイル先頭に TOC コメント

**ユーザー操作（レイアウトだけでなく挙動）:**
- TOP:「提訴する」タップ → INTAKE
- INTAKE: チャット気泡をタップで進行 → 事件名表示 → JUDGE_SELECT
- HEARING_A/B:「次へ」で気泡表示 → サマリー承認タップ
- HANDOFF:「準備OK」→ HEARING_B
- TRIAL:「次へ」または自動で発話表示
- VERDICT: タイプライター → 判決カード
- CLOSING: サマリーカード＋グレーアウトの控訴ボタン

シナリオ本文はまだ入れない。useState のみ。1ファイルに収める。
```

### Step 2: 中身（シナリオデータ接続）

```
Step 2 — 台本データをすべて接続。

Step 1 の骨組み・ステートマシン・レイアウト・デザイントークンは**そのまま維持**。

SCENARIO オブジェクトを追加（当事者、事件名、聞き取り全ターン、サマリー、法廷弁論、判決文を CLI ログから verbatim で）。

**画面ごとの挙動:**
1. TOP: アプリ名「家庭内裁判所」、説明、免責、「提訴する」
2. INTAKE: 書記官挨拶 → 申立人の発言気泡 →「受理しました。事件名を起案しています…」→ 事件名を演出付きで表示
3. JUDGE_SELECT: 5人ルーレット → 巌谷で停止 → 紹介カード＋開廷宣言
4. HEARING_A: 美咲の聞き取りを順次再生。進捗表示。サマリーカード。「この内容でよろしいですか？」→ 承認
5. HANDOFF: プライバシー説明＋「準備OK」
6. HEARING_B: 健太も同様
7. TRIAL: 観戦モード。発話者名＋役割色＋アバタースロット付きで順次表示
8. VERDICT/CLOSING: 判決全文 → サマリーカード（75:25 バー、提言、裁判官名）
9. ESCALATION: 参考画面（フッターメニュー「エスカレーション画面を見る」から）

日本語はシナリオの原文をそのまま使用。LLM シミュレーションなし。
```

### Step 3: 演出（アニメーション）

```
Step 3 — プレゼン用アニメーションを追加。**コンテンツと遷移は変更しない。**

- JUDGE_SELECT: スロットマシン／ルーレット（1.5〜2秒）、減速、巌谷でハイライト＋決め台詞
- HEARING: 気泡のフェード／スライドイン。代理人メッセージ間に入力中インジケータ（0.5秒）
- TRIAL: 発話が話者側からスライドイン。法廷風の背景パターン
- VERDICT: タイプライター（スキップボタン付き）。割合バー 0→75% / 0→25%。提言カードを順次表示
- 事件名: INTAKE と判決カードで儀式的な表示アニメ

長文シーケンスに「スキップ」ボタンを追加（ライブデモ用）。Tailwind + lucide-react のみ。
```

### Step 4: 画像スロット

```
Step 4 — JUDGE_IMAGES を配線。**レイアウトとシナリオ文は変更しない。**

ファイル先頭:
const JUDGE_IMAGES = { judge_ooka: null, judge_logica: null, judge_mitsuko: null, judge_iwaya: null, judge_luna: null };

<JudgeAvatar id={...} size={...} /> コンポーネント:
- JUDGE_IMAGES[id] あり → <img> 表示
- null → 色付き円＋最初の漢字（大/ロ/観/巌/ル）

使用箇所: ルーレット、紹介カード、法廷発話行、判決ヘッダー。

**似顔絵イラストは生成しない。** 後から URL/data URI を貼る行を教えてください。
```

### Step 5: QA＆引き渡し

```
Step 5 — 最終 QA。

チェックリストで確認:
□ TOP → CLOSING まで詰まらず完走
□ 日本語がシナリオと一致（省略なし）
□ 発話者ごとの役割色が正しい
□ 全画面に免責表示
□ 375px 幅で横スクロールなし
□ JUDGE_IMAGES が null と URL の両方で動作
□ localStorage / fetch / 外部 API なし
□ 控訴ボタンはグレーアウト「もう一度別の裁判官で裁いてもらう」
□ エスカレーション画面に到達可能
□ 判決のスキップボタンが動作

失敗箇所は**最小差分**で修正。先頭に「デモ操作ガイド」コメント（5分プレゼンの推奨タップ順）を追加。
```
