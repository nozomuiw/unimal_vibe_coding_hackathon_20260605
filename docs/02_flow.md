# 02. 体験フロー仕様

## 全体像（状態遷移）

```
[INTAKE 事由発生]
      │ Aさんが事象を説明し終える
      ▼
[JUDGE_SELECT 裁判官選定]
      │ ランダム選定 → 裁判官の登場演出
      ▼
[HEARING_A 聞き取り: Aさん]
      │ A代理人が「聞き取り完了」と判断 or 上限ターン到達
      ▼
[HANDOFF 端末受け渡し画面]
      │ Bさんが「準備OK」を押す
      ▼
[HEARING_B 聞き取り: Bさん]
      │ B代理人が「聞き取り完了」と判断 or 上限ターン到達
      ▼
[TRIAL 代理戦争] ◄────────────────────┐
      │                               │
      ├─ 裁判官「判定可能」→ [VERDICT] │
      │                               │
      └─ 裁判官「情報不足」            │
            ▼                         │
      [RE_HEARING 追加聞き取り] ──────┘
      （最大2回まで。超えたら強制的に VERDICT へ）

[VERDICT 判決]
      │ 判決文の表示 → 双方への説諭
      ▼
[CLOSING 閉廷]
```

どのフェーズでもセーフティ検知（[01_concept.md](01_concept.md) 対象外領域）に該当した場合は `ESCALATION` 状態に遷移し、裁判を中断して窓口案内を表示する。

---

## Phase 1: 話し合い事由発生（INTAKE）

**画面の主役: Aさん / 対話相手: 書記官（システム案内役）**

1. トップ画面で「提訴する」を押すと、書記官役のシステムメッセージが手続きを案内する
2. Aさんが事象を自由記述で説明する（例:「先週、夫が相談なしに10万円のキーボードを買った」）
3. システムが事案のタイトル（事件名）を生成して確認する
   - 例:「令和8年（家）第1号 無断高額キーボード購入事件」
   - 事件名の大仰さはエンタメ装置として重要
4. Bさん（相手方）の呼び名をAさんが入力する

**完了条件**: 事象の説明と事件名の確定。

## Phase 2: 裁判官選定（JUDGE_SELECT）

1. 5人の裁判官プールから**ランダムに1人**を選定する（キャラクター詳細は [03_characters.md](03_characters.md)）
2. ルーレット風の演出ののち、裁判官が自己紹介と開廷宣言を行う
   - ここで裁判官の口調・性格を体験者に印象づける。「この裁判官に当たったか…」という感情がゲーム性の起点になる
3. 裁判官は以後、判決まで固定（途中交代なし）

## Phase 3: 聞き取りフェーズ（HEARING_A / HEARING_B）

**画面の主役: Aさん（次いでBさん） / 対話相手: 各自の代理人AI**

### A代理人によるAさんへのヒアリング

1. A代理人が共感的な姿勢で自己紹介し、INTAKE の内容を踏まえて深掘り質問を行う
2. ヒアリングの観点（代理人が内部的に埋めるべき項目）:
   - **事実**: いつ・どこで・何があったか。頻度・金額など定量情報
   - **経緯**: 過去の合意・約束・前例の有無
   - **感情**: 何が一番嫌だったか、どう感じたか
   - **要望**: 相手にどうしてほしいか、どうなれば解決か
3. 代理人は毎ターン内部で「聞き取り完了か」を判定する（構造化出力。[04_prototype_spec.md](04_prototype_spec.md) 参照）
4. 完了条件: 上記4観点が埋まった、かつ代理人が「これ以上聞いても新情報が出ない」と判断
5. **上限: 10往復**。到達したら現状の情報で完了とする（ユーザーを疲れさせない）
6. 完了時、代理人が「あなたの言い分はこう理解しました」と**聞き取りサマリー**を提示し、Aさんが承認する（修正可）

### 端末受け渡し（HANDOFF）

1. 「端末をBさんに渡してください」画面を表示
2. **プライバシー設計**: Aさんのヒアリング内容（生の対話ログ）はBさんには見せない。Bさんに見えるのは事件名と「Aさんから提訴があった」事実のみ
   - 各自が代理人に本音を話せることが聞き取りの質を担保する
   - 法廷フェーズ（代理戦争）以降は双方に公開される。代理人は法廷に出す主張を、依頼人が承認したサマリーの範囲から構成する
3. Bさんが「準備OK」を押すと HEARING_B が開始

### B代理人によるBさんへのヒアリング

1. B代理人がBさんに事件の概要（事件名とAさんの主張の要旨）を伝え、Bさんの言い分を聞き取る
2. 以降の流れ・完了条件・上限は HEARING_A と同じ
3. B代理人もBさんの**味方**である。Aさんの主張を代弁するのではなく、Bさんの反論・事情・言い分を引き出す

## Phase 4: 代理戦争（TRIAL）

**画面の主役: 双方（一緒に観戦） / 登場: A代理人・B代理人・裁判官**

ここからは人間は入力しない。**観戦フェーズ**である。3者のAIによる弁論が劇場形式でチャット画面に流れる。

### 進行（1ラウンド）

```
裁判官: 開廷宣言・争点の整理
  ↓
A代理人: 主張陳述（Aさんの言い分を社会通念に接続して弁護）
  ↓
B代理人: 反論＋Bさん側の主張陳述
  ↓
A代理人: 再反論（1回）
  ↓
B代理人: 再反論（1回）
  ↓
裁判官: ラウンド総括 → 判定可否の宣言
```

- 各代理人は基本的に該当人の弁護をする。ただし**捏造は禁止**: 聞き取りで得た情報の範囲でのみ主張を構成する
- 裁判官はラウンド終了時に内部で判定する（構造化出力）:
  - `verdict_possible: true` → Phase 5（判決）へ
  - `verdict_possible: false` → 不足している論点と、各当事者への**追加質問リスト**を出力し、RE_HEARING へ

### 再聞き取り（RE_HEARING）

1. 裁判官の追加質問を各代理人が持ち帰り、該当者にだけ質問する（A向け質問→Aさん、B向け質問→Bさん。端末受け渡しを挟む）
2. 追加聞き取りは**1回あたり最大3往復**の短いセッション
3. 完了したら TRIAL に戻り、新情報を踏まえて弁論を再開する

### ループ上限

- 再聞き取りは**最大2回**まで
- 2回を超えても判定できない場合、裁判官は「現有の情報で判決を下す」と宣言して強制的に Phase 5 へ進む（無限ループ防止。「証拠不十分な点は判決文で言及する」という形で処理する）

## Phase 5: 判決（VERDICT）

**画面の主役: 双方 / 語り手: 裁判官**

判決文は以下の構成で、裁判官のキャラクター口調により出力される:

1. **主文**: どちらの言い分に正当性があるか。割合で示す（例:「Aさん 70 : Bさん 30」）
   - 100:0 は原則出さない。どちらにも言い分はある、が基本姿勢
2. **理由**: 社会通念上の根拠。双方の主張のどこを認め、どこを認めなかったか
3. **今後への提言**: 同じ争いを繰り返さないための具体的な行動提案（例:「2万円以上の買い物は事前にLINEで一報を入れる」）
4. **双方への説諭**: 勝った側にも負けた側にも、裁判官のキャラクターを通じた言葉をかける。負けた側の感情のケアをここで行う

### 閉廷（CLOSING）

- 閉廷宣言ののち、判決サマリーカード（事件名・主文・提言）を表示する
- 「もう一度別の裁判官で裁いてもらう」ボタン（控訴ごっこ）はプロトタイプでは将来課題としてUI上はグレーアウト表示のみ

## フェーズごとの入出力一覧

| フェーズ | 入力する人 | 対話するAI | 他方に見えるか |
|---|---|---|---|
| INTAKE | Aさん | 書記官（システム） | 事件名のみ |
| JUDGE_SELECT | （なし） | 裁判官 | 公開 |
| HEARING_A | Aさん | A代理人 | **非公開**（承認済みサマリーのみ法廷へ） |
| HEARING_B | Bさん | B代理人 | **非公開**（同上） |
| TRIAL | （観戦のみ） | A代理人・B代理人・裁判官 | 公開 |
| RE_HEARING | 該当者 | 該当者の代理人 | **非公開** |
| VERDICT / CLOSING | （観戦のみ） | 裁判官 | 公開 |

## ターン数の上限まとめ（疲労・コスト管理）

| 項目 | 上限 |
|---|---|
| 聞き取り（初回・各人） | 10往復 |
| 再聞き取り（1回あたり・各人） | 3往復 |
| 再聞き取りの回数 | 2回 |
| 代理戦争のラウンド数 | 最大3ラウンド（初回 + 再聞き取り後×2） |

想定所要時間: **1セッション 20〜30分**（聞き取り各5〜8分、代理戦争・判決の観戦5〜10分）。
